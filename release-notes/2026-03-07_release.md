# Release Notes — March 7, 2026

## New Features

### Core API Usage Metrics Sync
- Added a manual **"Sync Usage"** button in the Acquisition List page header (editors only).
- Clicking it opens a dialog where an editor can paste a Core API bearer token to authenticate the sync.
- The sync fetches usage data from `core-api.aisoftware.com/v3/rpc/data_metrics/org_summary` and matches clients by `orgId` (from `sfdc.Capacity_Org_ID__c`).
- For each matched client, individual usage boolean flags are set to `true` if the corresponding metric total is greater than 1:
  - `usageActiveUsers`, `usageAuthActiveUsers`, `usageAutomations`, `usageConversations`, `usageInquiries`, `usageTickets`, `usageVoiceMinutes`, `usageWorkflows`
- `activeInConsole` is set to `true` if **two or more** of the above usage flags are `true` (defined as "meaningful activity").
- After the sync, `clientActiveCount` on `AcquisitionProgress` is updated for all acquisitions where `manualSync = false` and `clientMetricsApplicable = true`.
- `usageSyncedAt` is recorded on each updated `AcquisitionClientCount` row.
- New API route: `POST /api/sync/usage` (editor session authenticated, bearer token passed in request body).

### Acquisition Tracker — Client Migration Tooltips
- Hovering over the **Clients With Access to Console** progress bar now shows a tooltip with counts for clients with access, without access, and the total.
- Hovering over the **Clients Active in the Console** progress bar now shows a tooltip with counts for active clients, not-yet-active clients, and the total.
- Consistent with the existing Functionality in Console epic breakdown tooltip.

### Manual Sync & Client Metrics Flag Enforcement
- The Vitally and Usage syncs now fully respect both the `manualSync` and `clientMetricsApplicable` flags on `AcquisitionProgress`.
- If `manualSync = true` **or** `clientMetricsApplicable = false`, the sync will still update `AcquisitionClientCount` records but will **skip** rolling up `clientCountTotal`, `clientAccessCount`, and `clientActiveCount` to `AcquisitionProgress`.

### Native PowerPoint Export — Acquisition Tracker Slides
- The PowerPoint export now generates **fully native slides** for Acquisition Tracker entries using `pptxgenjs` text boxes, shapes, and progress bars — no longer relying on screenshots for these slides.
- One slide per acquisition, with layout closely matching the Acquisition Tracker UI:
  - Navy header bar with "Acquisition Tracker" label
  - Acquisition name (uppercase, bold) with colour accent dot
  - Disposition badge (outline style, colour-coded)
  - Description text
  - Overall Integration Progress label, percentage, and bar
  - **Technical Integration** panel: Dev Platform card (Connected/Not Connected badge) and Functionality in Console card (segmented progress bar, legend, epic counts)
  - **Client Migrations** panel: Clients With Access and Clients Active cards (progress bars and counts; "Not Applicable" badge when `clientMetricsApplicable = false`)
  - Integration Overview section (when present) below the panels
- Generation is handled server-side via `POST /api/export/pptx` to avoid Node.js module conflicts with client-side bundling.

### Customer Roadmap Download
- Added a **"Download Customer Roadmap"** button to the standard roadmap view page, available to all users.
- Downloads `Capacity_2026_Product_Roadmap.pdf` directly from the browser.

## Improvements

### PowerPoint Export — Visual Refinements
- **Rounded corners:** All containers (step cards, panel backgrounds, progress bars, badges, legend dots) use `roundRect` shapes. Header bars intentionally remain square-edged.
- **Larger fonts:** Minimum body text size is 12pt. Section labels ("Overall Integration Progress", "Integration Overview") are 16pt bold. Step card titles are 13pt. Panel titles are 12pt bold.
- **Dynamic card heights:** Step card heights are computed from their actual content rows, eliminating empty whitespace in cards that contain fewer elements (e.g. Dev Platform has no progress bar).
- **Progress bar layout:** The overall progress bar is shorter horizontally to give the percentage number and "Combined Status" label their own dedicated column, preventing overlap.
- **Layout overflow fix:** Corrected a bug where the Functionality in Console card would overlap the Integration Overview section. Panel heights now correctly account for the taller of the two step-1 cards when computing vertical positions.

### PDF Export — Cover Page Logo
- The white Capacity wordmark logo is now included on the cover page of PDF exports. The SVG is rasterized to a canvas before embedding to ensure compatibility with jsPDF.

### PDF Export — Badge Alignment Fix
- Resolved a known `html2canvas` rendering issue where badge text appeared top-aligned inside the badge container.
- Fixed by removing `items-center` from the badge component's base classes and relying on equal vertical padding (`py-0.5`) and `leading-none` for centering — a rendering approach that `html2canvas` handles correctly.

### Technical Integration Page
- **Default view changed to "Group by Project"** — the page now opens in Project view instead of Acquisition view.
- **Wider first column** — the Acquisition/Project label column increased from 280px to 340px to accommodate longer names.
- **Smaller row label text** — parent row names (acquisition names in Acquisition view, project titles in Project view) are now `text-sm` to reduce visual weight.
- **Fixed transparent sticky header** — the "Project / Acquisition" corner cell now uses a solid background (`bg-card`) so that milestone column headers no longer show through it when scrolling horizontally.



### Vitally Sync — `orgId` Field Investigation
- Investigated incorrect `orgId` values being stored in `AcquisitionClientCount`. Found that `traits["sfdc.Capacity_Org_ID__c"]` contains unreliable data from Salesforce.
- Added a diagnostic `orgIdSample` field to the Vitally sync API response, returning the first 20 accounts with the raw field value, its type, and what was saved — to aid in identifying the correct source field.
- Updated `orgId` extraction to accept both `string` and `number` types from Vitally (previously only `string` was accepted, causing numeric org IDs to be saved as `null`).
- **Manually populated `orgId` values are now preserved:** the Vitally sync upsert no longer overwrites `orgId` with `null` — it only sets `orgId` if Vitally provides a non-null value, protecting any data entered directly in the database.

### Usage Sync — API Error Handling
- Fixed the usage sync API route returning HTTP 500 when the Core API itself returned an error, which caused the browser to show a generic failure with no details.
- The route now always returns HTTP 200 with the full result object (including the `errors` array), so error messages are surfaced in the sync result banner instead of being swallowed.

## Database / Schema Changes
- `AcquisitionClientCount`: added 9 new boolean columns — `usageActiveUsers`, `usageAuthActiveUsers`, `usageAutomations`, `usageConversations`, `usageInquiries`, `usageTickets`, `usageVoiceMinutes`, `usageWorkflows` (all default `false`) — and `usageSyncedAt` (nullable timestamp).
- `AcquisitionProgress`: added `clientMetricsApplicable` (boolean, default `true`).

---

## Configuration
- New environment variable: `CORE_API_TOKEN` — optional default bearer token for the usage sync (token can also be supplied manually per-sync via the UI dialog).
